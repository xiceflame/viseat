## 二、基线模式技术原理

### 2.1 设计动机

传统的双图对比方案存在以下问题：
- LLM 在不同角度下估算不一致
- 食物被翻动后难以准确对比
- 重新识别会产生累积误差

**基线模式解决方案**：
1. 用餐开始时建立食物基线（锁定食物列表和初始重量）
2. 后续只识别剩余量，通过 `消耗 = 基线 - 剩余` 计算
3. 保持食物名称一致性，避免重复识别误差

### 2.2 核心算法

```python
# 基线模式热量计算
for each baseline_food:
    remaining_weight = LLM识别当前剩余量(food_name)
    consumed_weight = baseline_weight - remaining_weight
    consumed_calories += calculate_nutrition(food_name, consumed_weight)
```

### 2.3 新食材发现机制

针对**盖浇饭**等上层遮挡底层的场景：

```
开始识别（俯视图）      翻动后（食物混合）
┌──────────────┐       ┌──────────────┐
│  牛肉  蔬菜  │       │ 米饭 牛肉    │
│  土豆  胡萝卜│  ->   │   蔬菜 土豆  │
│  ─────────── │       │     米饭     │
│  (米饭被遮挡) │       │ (米饭露出)   │
└──────────────┘       └──────────────┘

处理逻辑：
1. 检测到 "米饭" 不在基线中
2. 标记为新发现食材 (is_new: true)
3. 加入剩余列表，提示更新基线
4. 不算作已消耗（因为刚发现）
```

### 4.4 语义映射与别名系统

为解决 VLM 返回食材名与数据库标准名不一致的问题，系统构建了**多层级语义映射体系**：

```python
# 示例：VLM 返回 "steamed rice" → 系统映射到 "籼米"
别名映射层级:
├── 英文标准名映射: rice → 籼米
├── 中文别名映射: 白米饭 → 籼米  
├── 烹饪方式关联: steamed_rice → 籼米
└── 品牌/菜品映射: 老干妈炒饭 → 炒饭
```

**核心映射规则**：
- **英文到中文**: `beef` → `牛肉`, `chicken` → `鸡肉`
- **口语到标准**: `青菜` → `小白菜`, `米饭` → `籼米`
- **菜品到食材**: `红烧肉` → `猪肉(五花肉)`
- **品牌到通用**: `可口可乐` → `碳酸饮料`

## 六、核心技术创新与竞争优势

### 6.1 基线差分追踪算法（Baseline Differential Tracking）

**创新点**：业界首创的"基线锁定 + 差分计算"用餐追踪方案

| 传统方案 | VisEat 方案 |
|----------|-------------|
| 每次重新识别所有食物 | 首次建立基线，后续只识别剩余量 |
| 食物名称不一致导致误差累积 | 基线锁定保证名称一致性 |
| 重复识别增加计算开销 | 差分计算降低 60% 推理成本 |
| 无法处理食物被翻动的场景 | 自动发现新露出的食材 |

**核心公式**：`消耗量 = 基线量 - 当前剩余量`

### 6.2 动态基线自适应更新

针对真实用餐场景中食物状态变化的挑战：

- **遮挡食材发现**：自动检测翻动后露出的新食材（如盖浇饭底部的米饭）
- **运行时基线更新**：支持动态添加新发现的食材到基线
- **置信度加权**：根据识别置信度调整营养计算权重

### 6.3 健康档案驱动的个性化建议引擎
